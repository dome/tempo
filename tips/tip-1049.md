# TIP-1049: Admin Access Keys

## Abstract

This TIP adds an `admin` boolean flag to the `AuthorizedKey` storage struct and the `KeyRestrictions` configuration. When `admin` is `true`, the access key may authorize and revoke other access keys on behalf of the account, and is eligible for TIP-1020 signature verification. Non-admin access keys retain current behavior: they cannot manage keys and are invisible to TIP-1020.

## Motivation

Access keys today are strictly subordinate: they can sign transactions within their spending limits and call scopes, but they cannot authorize new keys or participate in onchain signature verification. This creates two practical problems:

1. **Cross-device key management.** A user with a passkey on their phone cannot authorize a second passkey on their laptop without a transaction from the root EOA key. In practice, multi-device setups require the root key to be available for every new device onboarding, which is inconvenient and fragile.

2. **Onchain signature verification.** TIP-1020 allows contracts to verify Tempo signatures, but there is no way for a contract to verify that a signature was produced by an authorized access key of a given account. Contracts that want to gate actions on "this account approved this message" (e.g. ERC-1271-style flows, gasless approvals, cross-chain authorizations) cannot use access keys for this purpose.

Adding an `admin` flag solves both problems with a single, conservative mechanism: admin keys can manage other keys and can be verified via TIP-1020, while non-admin keys remain restricted.

## Assumptions

- The root EOA key is always implicitly admin. The `admin` flag only applies to access keys registered in the AccountKeychain precompile.
- Admin keys inherit all existing access key restrictions (expiry, spending limits, call scopes). The `admin` flag is additive — it grants key-management and verification capabilities but does not bypass other restrictions.
- An admin key cannot grant the `admin` flag to keys it authorizes unless the admin key itself was authorized with that capability. This prevents privilege escalation: only the root key (or another admin key) can create admin keys.

# Specification

## Storage Layout

The `AuthorizedKey` packed struct gains a new `is_admin` boolean at byte 11:

```
Byte 0:     signature_type (u8)
Bytes 1–8:  expiry (u64, little-endian)
Byte 9:     enforce_limits (bool)
Byte 10:    is_revoked (bool)
Byte 11:    is_admin (bool)        ← NEW
```

This adds one byte to the existing packed slot. No new storage slots are introduced.

## Interface Changes

### `KeyRestrictions`

Add an `admin` field to the `KeyRestrictions` struct:

```solidity
struct KeyRestrictions {
    uint64 expiry;
    bool enforceLimits;
    TokenLimit[] limits;
    bool allowAnyCalls;
    CallScope[] allowedCalls;
    bool admin;               // NEW
}
```

### `KeyInfo`

Add an `isAdmin` field to the `KeyInfo` return struct:

```solidity
struct KeyInfo {
    SignatureType signatureType;
    address keyId;
    uint64 expiry;
    bool enforceLimits;
    bool isRevoked;
    bool isAdmin;             // NEW
}
```

### `KeyAuthorized` Event

Add `isAdmin` to the `KeyAuthorized` event:

```solidity
event KeyAuthorized(
    address indexed account,
    address indexed publicKey,
    uint8 signatureType,
    uint64 expiry,
    bool isAdmin                // NEW
);
```

## Behavioral Changes

### Key Authorization by Access Keys

Currently, the handler rejects any transaction where an access key attempts to authorize a different key (`AccessKeyCannotAuthorizeOtherKeys`). With this TIP:

- If the signing access key has `is_admin = true`, the transaction MAY include a `KeyAuthorization` for a different `key_id`. The `KeyAuthorization` must still be signed by the root account key.
- If the signing access key has `is_admin = false`, the existing restriction remains: the key can only authorize itself (same-tx auth+use).

### Key Revocation by Access Keys

Currently, only the root key can call `revokeKey`. With this TIP:

- Admin access keys MAY call `revokeKey` to revoke other access keys on the same account.
- Non-admin access keys MUST NOT call `revokeKey`.

A new `ensure_admin_key_caller` check is introduced, separate from `ensure_admin_caller`, that accepts admin access keys (identified via the `transaction_key` transient storage) but is scoped strictly to `authorizeKey` and `revokeKey`. Admin access keys MUST NOT reach sibling-key mutators (`updateSpendingLimit`, `setAllowedCalls`, `removeAllowedCalls`) — those remain root-key-only via the existing `ensure_admin_caller` gate.

The T2 `tx_origin == msg_sender` guard MUST be preserved for the admin key path: an admin key used from an intermediate contract (not as the direct transaction signer) MUST be rejected to prevent confused-deputy attacks.

### TIP-1020 Signature Verification

Currently, TIP-1020's `recover` and `verify` functions reject keychain-type signatures (`0x03`, `0x04` prefixes) with `InvalidFormat`. With this TIP:

- V1 keychain signatures (`0x03` prefix) MUST be rejected unconditionally with `InvalidFormat()`. Only V2 keychain signatures (`0x04` prefix) are accepted, consistent with the T1C fix (`validate_version`).
- TIP-1020 verification for V2 keychain signatures is accepted when the recovered key has `is_admin = true` in the AccountKeychain for the given account.
- The verification flow becomes:
  1. If the prefix byte is `0x03`, MUST revert with `InvalidFormat()`.
  2. Decode the `0x04` keychain signature to extract the inner signature, key ID, and `user_address`.
  3. Look up the key in AccountKeychain for the claimed account. Verify the key exists, is not revoked, is not expired, has `is_admin = true`, and its stored `signature_type` matches the inner signature's type.
  4. Compute `effective_hash = keccak256(0x04 || hash || user_address)`. Verify the inner signature against `effective_hash` and the key ID.
- If the key is not admin, verification MUST revert.
- `recover(hash, signature)` MUST return `user_address` (the root account), not the `key_id`. `verify(signer, hash, signature)` MUST return `true` iff `signer == user_address`.

#### Gas

The keychain-admin verification path adds stateful `AccountKeychain` lookups. The TIP-1020 gas table is extended:

| Signature type | Gas cost |
|---|---|
| Keychain-admin (V2, `0x04`) | Base crypto cost (per inner type) + 2100 (cold SLOAD for key lookup) |

### Admin Key Privilege Propagation

An admin access key MAY authorize new keys with `admin = true`, enabling delegation chains. This is intentional: it allows multi-device setups where any trusted device can onboard new devices without requiring the root key to be *interactively available* — the admin key signs the transaction that carries the `KeyAuthorization`, but the `KeyAuthorization` itself must still be signed by the root account key (which may have been pre-signed or delegated via a custodial flow).

Revocation of an admin key does NOT automatically revoke keys that it authorized. Each key is independently tracked; cascading revocation is out of scope for this TIP. Accounts that need cascading revocation should revoke downstream keys explicitly.

## Transaction Encoding

The `KeyAuthorization` RLP encoding is bumped to V2 to carry the `admin` field unambiguously. The version is indicated by a new leading byte:

```
V1 (existing): [chain_id, key_type, key_id, expiry?, limits?, allowed_calls?]
V2 (this TIP):  [0x02, chain_id, key_type, key_id, expiry?, limits?, allowed_calls?, admin]
```

- `admin` is encoded as `u8`: `0x00` = non-admin, `0x01` = admin. `Option<bool>` is NOT used because `bool::false` decodes from `0x80` (the RLP `None` sentinel), which is ambiguous under `#[rlp(trailing(canonical))]`.
- V1 authorizations remain valid and default to `admin = false`.
- V2 authorizations MUST include the `admin` field explicitly (not trailing-optional).

# Invariants

1. **Admin-only key management.** A non-admin access key MUST NOT authorize or revoke other keys. Transactions that violate this MUST be rejected with `AccessKeyCannotAuthorizeOtherKeys`.

2. **Root key equivalence for admin operations.** Admin access keys MUST be subject to the same validation as root key operations for `authorizeKey` and `revokeKey`: the `KeyAuthorization` must be signed by the root account, and all existing checks (expiry, revocation, signature type) apply. The T2 `tx_origin == msg_sender` guard MUST be enforced for admin key callers.

3. **No implicit privilege escalation.** The `admin` flag does not bypass spending limits, call scopes, or expiry. An admin key with a 1-hour expiry expires after 1 hour regardless of admin status. Admin keys MUST NOT reach sibling-key mutators (`updateSpendingLimit`, `setAllowedCalls`, `removeAllowedCalls`).

4. **TIP-1020 admin gate.** Keychain signature verification via TIP-1020 MUST succeed only for V2 (`0x04`) keychain signatures where the recovered key has `is_admin = true`. V1 (`0x03`) keychain signatures MUST be rejected with `InvalidFormat()`. Non-admin keys MUST cause verification to revert.

5. **Signature-type binding.** TIP-1020 keychain verification step 3 MUST require the key's stored `signature_type` to match the inner signature's type. Mismatches MUST revert.

6. **TIP-1020 return semantics.** `recover` MUST return `user_address` (root account). `verify(signer, …)` MUST return `true` iff `signer == user_address`.

7. **Backward compatibility.** Existing access keys (authorized before this TIP activates) default to `is_admin = false`. Their behavior is unchanged. V1 `KeyAuthorization` RLP encodings remain valid and default to non-admin.

8. **No cascading revocation.** Revoking an admin key MUST NOT automatically revoke keys that were authorized while that admin key was active. Each key's revocation status is independent.
