# TIP-1049: Admin Access Keys

## Abstract

This TIP adds an `admin` boolean flag to the `AuthorizedKey` storage struct and the `KeyRestrictions` configuration. When `admin` is `true`, the access key may authorize and revoke other access keys on behalf of the account, and is eligible for TIP-1020 signature verification. Non-admin access keys retain current behavior: they cannot manage keys and are invisible to TIP-1020.

## Motivation

Access keys today are strictly subordinate: they can sign transactions within their spending limits and call scopes, but they cannot authorize new keys or participate in onchain signature verification. This creates two practical problems:

1. **Cross-device key management.** A user with a passkey on their phone cannot authorize a second passkey on their laptop without a transaction from the root EOA key. In practice, multi-device setups require the root key to be available for every new device onboarding, which is inconvenient and fragile.

2. **Onchain signature verification.** TIP-1020 allows contracts to verify Tempo signatures, but there is no way for a contract to verify that a signature was produced by an authorized access key of a given account. Contracts that want to gate actions on "this account approved this message" (e.g. ERC-1271-style flows, gasless approvals, cross-chain authorizations) cannot use access keys for this purpose.

Adding an `admin` flag solves both problems with a single, conservative mechanism: admin keys can manage other keys and can be verified via TIP-1020, while non-admin keys remain restricted.

## Assumptions

- The root EOA key is always implicitly admin. The `admin` flag only applies to access keys registered in the AccountKeychain precompile.
- Admin keys inherit all existing access key restrictions (expiry, spending limits, call scopes). The `admin` flag is additive — it grants key-management and verification capabilities but does not bypass other restrictions.
- An admin key cannot grant the `admin` flag to keys it authorizes unless the admin key itself was authorized with that capability. This prevents privilege escalation: only the root key (or another admin key) can create admin keys.

# Specification

## Storage Layout

The `AuthorizedKey` packed struct gains a new `is_admin` boolean at byte 11:

```
Byte 0:     signature_type (u8)
Bytes 1–8:  expiry (u64, little-endian)
Byte 9:     enforce_limits (bool)
Byte 10:    is_revoked (bool)
Byte 11:    is_admin (bool)        ← NEW
Bytes 12–31: authorized_by (address) ← NEW
```

`authorized_by` records which key authorized this key. For keys authorized by the root EOA, this is `address(0)`. This enables authorizer provenance queries and cascading revocation without event indexing.

## Interface Changes

### `KeyRestrictions`

Add an `admin` field to the `KeyRestrictions` struct:

```solidity
struct KeyRestrictions {
    uint64 expiry;
    bool enforceLimits;
    TokenLimit[] limits;
    bool allowAnyCalls;
    CallScope[] allowedCalls;
    bool admin;               // NEW
}
```

### `KeyInfo`

Add an `isAdmin` field to the `KeyInfo` return struct:

```solidity
struct KeyInfo {
    SignatureType signatureType;
    address keyId;
    uint64 expiry;
    bool enforceLimits;
    bool isRevoked;
    bool isAdmin;             // NEW
    address authorizedBy;     // NEW — which key authorized this key (address(0) = root)
}
```

### `KeyAuthorized` Event

Add `isAdmin` to the `KeyAuthorized` event:

```solidity
event KeyAuthorized(
    address indexed account,
    address indexed publicKey,
    uint8 signatureType,
    uint64 expiry,
    bool isAdmin,               // NEW
    address authorizedBy        // NEW
);
```

## Behavioral Changes

### Key Authorization by Access Keys

Currently, the handler rejects any transaction where an access key attempts to authorize a different key (`AccessKeyCannotAuthorizeOtherKeys`). With this TIP:

- If the signing access key has `is_admin = true`, the transaction MAY include a `KeyAuthorization` for a different `key_id`.
- If the signing access key has `is_admin = false`, the existing restriction remains: the key can only authorize itself (same-tx auth+use).

### Offchain Key Authorization (ERC-1271-style)

An admin access key MAY authorize a new key via an offchain signature, without requiring a transaction signed by the root EOA key. The flow:

1. Admin key A produces an ERC-1271-style signature over a `KeyAuthorization` message for new key B.
2. Any relayer submits a transaction calling `authorizeKeyWithSig(account, keyAuthorization, signature)` on the AccountKeychain precompile.
3. The precompile verifies: (a) the signature recovers to a key that is admin on the account, (b) the `KeyAuthorization` is well-formed, (c) a replay-resistant nonce is consumed.
4. The new key is authorized with `authorized_by` set to the signing admin key.

This enables cross-device approval: passkey A on iOS can sign an offchain approval for passkey B on Android, and B (or a third-party relayer) submits the authorization transaction. The root EOA key does not need to be available.

```solidity
function authorizeKeyWithSig(
    address account,
    KeyAuthorization calldata auth,
    bytes calldata adminSignature
) external;
```

The signature MUST cover `keccak256(abi.encode(account, auth, nonce, block.chainid))` to prevent replay across accounts and chains.

### Key Revocation by Access Keys

Currently, only the root key can call `revokeKey`. With this TIP:

- Admin access keys MAY call `revokeKey` to revoke other access keys on the same account.
- Non-admin access keys MUST NOT call `revokeKey`.

The `ensure_admin_caller` check in the AccountKeychain precompile is updated to also accept admin access keys (identified via the `transaction_key` transient storage).

### Expiry Extension

Currently, `expiry` is set once at authorization time and cannot be changed. This forces re-authorization from scratch when a key expires, which is disruptive for long-lived devices.

Admin keys (or the root key) MAY call `extendExpiry` to update the expiry of an existing key:

```solidity
function extendExpiry(address account, address keyId, uint64 newExpiry) external;
```

- `newExpiry` MUST be strictly greater than the current expiry. Shortening expiry is not permitted (use `revokeKey` instead).
- The caller MUST be the root key or an admin access key on the account.
- The target key MUST NOT be revoked.

This allows devices to rotate/renew their keys without requiring a fresh authorization from the root key.

### Cascading Revocation

Revoking an admin key MAY optionally cascade to all keys it authorized. The `revokeKey` function gains an optional `cascade` flag:

```solidity
function revokeKey(address account, address keyId, bool cascade) external;
```

When `cascade` is `true`:

1. The target key is revoked.
2. All keys where `authorized_by == keyId` are also revoked (recursively).
3. A `KeyRevoked` event is emitted for each revoked key.

When `cascade` is `false` (or omitted for backward compatibility), only the target key is revoked. Downstream keys remain active.

This enables clean device offboarding: revoking a device's admin key with `cascade = true` removes the entire sub-tree of keys it authorized in a single transaction.

### TIP-1020 Signature Verification

Currently, TIP-1020's `recover` and `verify` functions reject keychain-type signatures (`0x03`, `0x04` prefixes) with `InvalidFormat`. With this TIP:

- TIP-1020 verification for keychain signatures is accepted when the recovered key has `is_admin = true` in the AccountKeychain for the given account.
- The verification flow becomes:
  1. Decode the keychain signature to extract the inner signature and key ID.
  2. Look up the key in AccountKeychain for the claimed account.
  3. Verify the key exists, is not revoked, is not expired, and has `is_admin = true`.
  4. Verify the inner signature against the key ID using standard TIP-1020 logic.
- If the key is not admin, verification MUST revert.

### Admin Key Privilege Propagation

An admin access key MAY authorize new keys with `admin = true`, enabling delegation chains. This is intentional: it allows multi-device setups where any trusted device can onboard new devices without the root key.

### `allKeys` Query

A new read-only function returns all active keys for an account:

```solidity
function allKeys(address account) external view returns (KeyInfo[] memory);
```

This returns all non-revoked, non-expired keys for the account, including the `authorizedBy` field. Paired devices can render the full key fleet without indexing `KeyAuthorized` events.

## Transaction Encoding

The `KeyAuthorization` RLP encoding gains an optional trailing `admin` field:

```
RLP: [chain_id, key_type, key_id, expiry?, limits?, allowed_calls?, admin?]
```

- `None` (omitted or `0x80`) = non-admin key (default)
- `Some(true)` / `0x01` = admin key

This is backward compatible: existing key authorizations without the field default to non-admin.

# Invariants

1. **Admin-only key management.** A non-admin access key MUST NOT authorize or revoke other keys. Transactions that violate this MUST be rejected with `AccessKeyCannotAuthorizeOtherKeys`.

2. **Root key equivalence for admin operations.** Admin access keys MUST be subject to the same validation as root key operations for `authorizeKey`, `authorizeKeyWithSig`, `revokeKey`, and `extendExpiry`: all existing checks (expiry, revocation, signature type) apply.

3. **No implicit privilege escalation.** The `admin` flag does not bypass spending limits, call scopes, or expiry. An admin key with a 1-hour expiry expires after 1 hour regardless of admin status.

4. **TIP-1020 admin gate.** Keychain signature verification via TIP-1020 MUST succeed only for keys with `is_admin = true`. Non-admin keys MUST cause verification to revert.

5. **Backward compatibility.** Existing access keys (authorized before this TIP activates) default to `is_admin = false`. Their behavior is unchanged.

6. **Cascading revocation correctness.** When `revokeKey` is called with `cascade = true`, all keys in the `authorized_by` sub-tree MUST be revoked recursively. When `cascade` is `false`, only the target key is revoked.

7. **Offchain authorization replay resistance.** Each `authorizeKeyWithSig` call MUST consume a unique nonce. Replaying the same signature MUST revert.

8. **Expiry monotonicity.** `extendExpiry` MUST only accept a `newExpiry` strictly greater than the current expiry. It MUST revert on revoked keys.

9. **Authorizer provenance.** `authorized_by` MUST be set to the key that performed the authorization (or `address(0)` for root). It MUST NOT be mutable after authorization.
